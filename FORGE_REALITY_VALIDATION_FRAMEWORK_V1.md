```
DOCUMENTED KNOWLEDGE
  └── FORGE Generator ──→ Generated Program
                              │

                    ┌──────────────────────┐

                              │

                    VALIDATED COACHING SYSTEM
```

# PART 1 — COACH REVIEW SCORECARD

## Purpose

Provide a standardised form for S&C coaches to evaluate generated programs.
Each criterion scored 1–10 with exact behavioural anchors.
No methodology names. No theory. Only what a coach would actually see in the program.

## Scoring Scale

| Score | Meaning | Action |
|-------|---------|--------|
| 10 | Perfect — no change needed | Flag as reference output |
| 8–9 | Excellent — minor tweak only | Accept as-is |
| 6–7 | Acceptable — would deliver with notes | Coach adds 1–2 modifications |
| 4–5 | Needs work — multiple issues | Return for regeneration |
| 2–3 | Poor — fundamental problems | Reject. Flag blueprint/exercise gap |
| 1 | Dangerous — would not deliver | Critical failure. Escalate to rulebook |

---

## Criterion 1: Exercise Selection

*Does the program choose exercises appropriate to the athlete's level, equipment, injury history, and training context?*

| Score | Behavioural Anchor | 
|-------|-------------------|
| 10 | Every exercise is the best choice for the athlete at this moment. Progression/regression correctly applied. No better option exists in the family. |
| 9 | One exercise could be marginally improved but current choice is still valid. |
| 8 | All exercises are correct for level/equipment/injury. None are wrong, but 1–2 are suboptimal (e.g., chose L2 when L3 is available and appropriate). |
| 7 | Exercises are broadly correct. 1–2 are too easy or too hard for the athlete's current level. |
| 6 | One exercise is inappropriate (wrong difficulty, equipment mismatch, or injury conflict). Others are fine. |
| 5 | 2–3 exercises are inappropriate. Or one exercise is dangerous for the athlete profile. |
| 4 | Multiple exercises are at the wrong level. Equipment mismatches throughout. |
| 3 | Most exercises are inappropriate for the athlete. Selection appears random within family. |
| 2 | Exercises are systematically wrong (all too hard, wrong families for the goal, ignoring injuries). |
| 1 | Exercise selection is unsafe. Would not deliver this session. |

**Pass threshold:** ≥ 6

| 1 | Session is structurally incoherent. Cannot be fixed by reordering — needs redesign. |

## Comparison Metrics

### Metric 1: Family Frequency Distribution

*Does FORGE select exercise families at the same rate as real-world programs?*

```
FOR EACH sport:
  Examine N real-world programs from that sport
  Calculate: frequency of each family (DLKD, DLHD, HPush, HPull, ...)
  Compare to: FORGE's family selection frequency for same sport profile

  Output: Family Frequency Deviation Score
    - Green (0–5% deviation): FORGE matches real-world
    - Amber (5–15% deviation): Investigate
    - Red (>15% deviation): FORGE is over/under-selecting a family
```

**Implementation:**
- Sample 10 real programs per sport (from FORGE_SOURCE_CATALOG Tier A sources)
- Extract family frequency distribution
- Run 100 FORGE generations for matching athlete profiles
- Compare distributions using chi-square or percentage deviation

### Metric 2: Session Ordering Compliance

*Does FORGE order session blocks the same way as real-world programs?*

```
FUNCTION compare_session_order(forge_session, real_program):
  forge_order = extract_block_order(forge_session)    # e.g. [Plyo, DLKD, HPush, HPull, Core]
  real_order  = extract_block_order(real_program)     # e.g. [Sprint, DLKD, HPush, HPull, Core]
  
  score = ORDER_SIMILARITY(forge_order, real_order)
  # Longest Common Subsequence / total blocks
  # 1.0 = identical order, 0.0 = completely different
  
  RETURN score
```

**Benchmarks:**
- LCS similarity ≥ 0.8: FORGE order matches real-world
- LCS similarity < 0.6: Order deviation — investigate blueprint slot_order

### Metric 3: Power Placement Concordance

*Do FORGE and real-world programs place power blocks at the same point in the session?*

```
FOR EACH pair (forge_session, real_program):
  forge_power_position = index of first power block (0-based)
  real_power_position = index of first power block
  
  IF forge_power_position == real_power_position:
    SCORE += 1  # Perfect match
  ELIF abs(forge_power_position - real_power_position) == 1:
    SCORE += 0.5  # Adjacent position
  ELSE:
    SCORE += 0  # Different position
  
  Normalise: score / total_comparisons
```

**Benchmark:** Concordance ≥ 0.9 = FORGE correctly places power

### Metric 4: Conditioning Placement Concordance

*Do FORGE and real-world programs place conditioning at the same point?*

- Same protocol: conditioning after strength = 1.0
- Adjacent: conditioning as final block (after core) or core after conditioning = 0.5
- Different: conditioning in the middle = 0.0

**Benchmark:** Concordance ≥ 0.85

### Metric 5: Progression Model Comparison

*Does FORGE's progression approach match real-world coaching practice?*

```
FOR EACH source:
  Extract: how exercises progress (reps-first? load-first? by RPE?)
  Compare to: FORGE's progression rules (law V041: reps before load)
  
  Score:
    Same progression priority: 1.0
    Different order but compatible: 0.5
    Contradictory: 0.0
```

**Benchmark:** Mean score ≥ 0.8 across all sources

# PART 3 — EDIT DISTANCE METRIC

## Purpose

Quantify how many changes a coach would make to a FORGE-generated program before delivering it.

Analogy: Levenshtein distance for text strings, but applied to training programs.

## Edit Operations

| Operation | Symbol | Weight | Description |
|-----------|--------|--------|-------------|
| Exercise Swap | S | 1.0 | Replace one exercise with another (same family, different exercise) |
| Exercise Remove | R | 1.0 | Remove an exercise entirely |
| Exercise Add | A | 1.5 | Insert a new exercise not in the generated program |
| Set Change | SC | 0.3 | Adjust set count (±1–2 sets) |
| Rep Change | RC | 0.3 | Adjust rep range |
| Rest Change | RST | 0.2 | Adjust rest interval |
| Load Adjustment | L | 0.5 | Change load prescription (RPE, %, or RIR) |
| Block Reorder | B | 2.0 | Reorder two training blocks |
| Conditioning Swap | CS | 1.0 | Replace conditioning protocol |
| Conditioning Parameter | CP | 0.5 | Adjust conditioning sets/rest/duration |

## Calculation

```
Coach Edit Distance = Σ(operation_count × weight)
```

### Interpretation

| CED Score | Meaning | Coach Time to Fix |
|-----------|---------|-------------------|
| 0 | Identical to what coach would write | 0 min |
| 1–3 | Minor tweaks | 2–5 min |
| 4–6 | Moderate revision | 5–10 min |
| 7–10 | Significant rewrite | 10–20 min |
| > 10 | Near-total rewrite | 20+ min |

### Acceptance Threshold

| Phase | Max Acceptable CED |
|-------|-------------------|
| V1 (current) | ≤ 10 |
| V1.1 target | ≤ 6 |
| V2 target | ≤ 3 |
| Production target | ≤ 1 |

## Edit Distance by Blueprint

Track CED separately for each blueprint type:

```
Blueprint 01 (Full Body Strength):   CED = 2.3  (N=10 coach reviews)
Blueprint 02 (Strength + Power):     CED = 3.1
Blueprint 03 (Strength + Cond):      CED = 4.2
...
Blueprint 12 (Return to Sport):      CED = 5.8
```

## Category Breakdown

When reporting CED, break down by category:

```
Total CED: 4.7
  Exercise swaps:    2 (× 1.0 = 2.0)
  Set/rep changes:   4 (× 0.3 = 1.2)
  Rest changes:      3 (× 0.2 = 0.6)
  Block reorders:    0 (× 2.0 = 0.0)
  Conditioning swap: 1 (× 1.0 = 1.0)
  Conditioning param: 1 (× 0.5 = 0.5)
  Load adjustments:  2 (× 0.5 = 1.0)
                        Total: 6.3
```

This breakdown tells FORGE *where* the gap is (exercise selection vs conditioning vs load assignment).

## Automated Edit Distance Estimator

An algorithm can estimate CED without human review by comparing to real-world program templates:

```
def estimate_edit_distance(generated: Session, template: Session) -> float:
    distance = 0.0
    
    # Compare exercise IDs per family
    for family in ALL_FAMILIES:
        gen_ex = generated.get_exercises_in_family(family)
        tmpl_ex = template.get_exercises_in_family(family)
        diff = set(gen_ex) - set(tmpl_ex)
        distance += len(diff) * 1.0  # Exercise swaps
    
    # Compare sets/reps for matching exercises
    for gen_block, tpl_block in zip(generated.blocks, template.blocks):
        if gen_block.exercise.id == tpl_block.exercise.id:
            if gen_block.sets != tpl_block.sets:
                distance += 0.3
            if gen_block.rep_range != tpl_block.rep_range:
                distance += 0.3
    
    # Compare conditioning
    if generated.conditioning and template.conditioning:
        if generated.conditioning.id != template.conditioning.id:
            distance += 1.0
    
    return distance
```

# PART 4 — BLUEPRINT VALIDATION

## Purpose

Evaluate every blueprint individually for credibility, completeness, applicability, and coach acceptance.

## Validation Criteria

### 1. Credibility (0–10)
Does this blueprint produce sessions that look like real coaching programs?

| Score | Meaning |
|-------|---------|
| 8–10 | Coaches would immediately recognise this as a legitimate program type |
| 5–7 | Would be accepted but may need justification |
| 0–4 | Coaches would question the approach |

Does the blueprint produce fully populated sessions (no empty slots, all required families present)?

| Score | Meaning |
|-------|---------|
| 8–10 | Every slot fills reliably across all equipment profiles |
| 5–7 | Occasional empty slots with specific equipment constraints |

How many athlete profiles can this blueprint serve?

| Score | Meaning |
|-------|---------|
| 8–10 | Serves ≥3 sports, ≥2 levels, ≥2 season phases |
| 5–7 | Serves 1–2 sports, 1–2 levels, specific phases |
| 0–4 | Only serves a niche combination |

Measured: What is the average Coach Edit Distance across 3 coach reviews per blueprint?

| Score | Meaning |

- **≥ 8.0:** Production-ready

### Validation Protocol Per Blueprint

```
FOR EACH blueprint:
  1. Generate N=10 sessions across different athlete profiles
     (vary: sport, level, season, equipment, injury)
  
  2. Run automated validation (12-point credibility check)
     → Track: pass rate, average score, common failures
  
  3. Run automated edit distance against template sessions
     → Track: average CED, category breakdown
  
  4. Submit to ≥2 S&C coaches for scorecard review
     → Track: average score per criterion, overall, comments
  
  5. Aggregate:
     credibility = scorecard average (criteria 1-5)
     completeness = % of sessions with all slots filled × 10
     applicability = number of valid athlete combinations / total possible
     coach_accept = (10 - average CED) clamped to [0, 10]
```

# PART 5 — PROGRAM ACCEPTANCE THRESHOLD

## Purpose

Define a single, objective classification for any generated program.

## Acceptance Levels

### Level 1: FAIL

**Definition:** Program cannot be delivered to an athlete.
**Criteria (any one):**
- Coach Scorecard total < 6.0
- 12-point credibility check < 8/10
- Edit Distance > 10
- Any safety-critical issue (dangerous exercise, contraindicated protocol)
- Empty slots (unfilled exercise families)
- Athlete-injury conflict detected

**Action:** Do not deliver. Flag for regeneration or escalation.
**FORGE Response:** Retry generation with tightened constraints (lower difficulty, substitute families, default to safer exercises). If 3 retries fail: produce a "cannot generate" report explaining why.

### Level 2: ACCEPTABLE

**Definition:** Program is structurally sound and safe, but a coach would modify 2–4 elements before delivering.
**Criteria (all):**
- Coach Scorecard 6.0–6.9
- OR CED 7–10
- 12-point check ≥ 8/10
- No safety issues
- No empty slots

**Action:** Deliver with "Coach Notes Required" flag. Coach reviews and adjusts.
**FORGE Response:** Attach specific notes about which elements are likely to need adjustment. Flag the session for improvement tracking.

### Level 3: GOOD

**Definition:** Program is coach-ready. A coach could deliver with confidence, making only minor adjustments.
**Criteria (all):**
- Coach Scorecard 7.0–8.4
- CED 4–6
- 12-point check ≥ 9/10
- No safety issues
- All slots filled

# PART 6 — 500 PROGRAM AUDIT PLAN

## Purpose

Run a systematic audit of 500 generated programs to establish baseline quality metrics, identify systematic weaknesses, and prioritise improvements.

## Audit Scope

| Dimension | Values | Combinations |
|-----------|--------|-------------|
| Blueprints | All 14 | 14 |
| Sports | rugby, american_football, tennis, badminton, basketball, volleyball, netball, squash, track_field, general | 10 |
| Athlete Levels | Beginner, Intermediate, Advanced | 3 |
| Season Phases | off_season, pre_season, in_season, transition | 4 |
| Equipment Profiles | Field Only, Basic Gym, Commercial Gym, Elite Facility | 4 |
| Injury Conditions | none, 1 injury, 2+ injuries, active injury (RTS) | 4 |
| Goals | general_strength, power, speed, conditioning, mass, return_to_sport, maintenance | 7 |

**Total theoretical combinations:** 14 × 10 × 3 × 4 × 4 × 4 × 7 = 188,160

**Sampling strategy:** Stratified random — ensure each blueprint appears at least 35 times. Weight by expected real-world frequency.

## Generation Script

```python
# audit_500.py — 500 Program Audit

import random
from forge.main import generate_program
from forge.models import (
    AthleteProfile, AthleteLevel, EquipmentProfile, SeasonPhase,
)
from forge.validator import verify_credibility, calculate_credibility_score
from forge.renderer import render_program


AUDIT_CONFIGS = []  # 500 athlete profiles

def build_audit_profiles():
    """Generate 500 athlete profiles covering the audit scope."""
    profiles = []
    
    sports = ["rugby", "basketball", "tennis", "track_field", "general",
              "american_football", "badminton", "volleyball", "football"]
    levels = [AthleteLevel.BEGINNER, AthleteLevel.INTERMEDIATE, AthleteLevel.ADVANCED]
    seasons = list(SeasonPhase)
    equipments = list(EquipmentProfile)
    goals = ["general_strength", "power", "speed", "conditioning",
             "mass", "return_to_sport", "power_maintenance"]
    injuries = [[], ["acl_left_2023"], ["low_back"], ["shoulder", "hamstring"]]
    
    # Ensure every blueprint is covered at least 35 times
    # Then fill remaining with weighted random distribution
    
    blueprint_weights = {i: 1 for i in range(1, 15)}
    # Weight common blueprints higher
    blueprint_weights[1] = 5  # Full Body Strength
    blueprint_weights[7] = 3  # Youth Foundation
    blueprint_weights[12] = 2  # Return to Sport
    
    for _ in range(500):
        profile = AthleteProfile(
            sport=random.choice(sports),
            training_age_years=random.uniform(0.5, 8.0),
            season_phase=random.choice(seasons),
            goal=random.choice(goals),
            equipment_profile=random.choice(equipments),
            athlete_level=random.choice(levels),
            technique_consistency=random.uniform(0.6, 1.0),
            injury_history=random.choice(injuries),
            fatigue_level=random.choice(["normal", "high"]),
        )
        profiles.append(profile)
    
    return profiles


def run_audit(profiles):
    """Generate and validate programs for all profiles."""
    results = []
    
    for i, profile in enumerate(profiles):
        program = generate_program(profile)
        check = verify_credibility(program.sessions[0], profile)
        score = calculate_credibility_score(check)
        
        result = {
            "index": i,
            "sport": profile.sport,
            "level": profile.athlete_level.value,
            "season": profile.season_phase.value,
            "equipment": profile.equipment_profile.value,
            "goal": profile.goal,
            "blueprint_name": program.blueprint_name,
            "credibility": score,
            "session_count": len(program.sessions),
            "pass": score >= 8.0,
            "failures": [k for k, v in check.items() if not v],
        }
        results.append(result)
        
        if (i + 1) % 50 == 0:
            print(f"  Audited {i + 1}/500...")
    
    return results
```

# PART 7 — FEEDBACK LOOP

## Purpose

Close the gap between generated programs and coach expectations.
Coach feedback must improve FORGE without changing the architecture.

## Principle

The architecture is stable (8-step pipeline: blueprint → level → equipment → slots → exercise → variables → validate → render). Feedback updates the **data** and **rules**, not the pipeline structure.

## What Can Change (Without Architecture Changes)

| Component | Change Mechanism | Effect | Example |
|-----------|-----------------|--------|---------|
| SELECTION_PRIORITIES | Reorder list | Different exercise selected first | Move Goblet Squat above Air Squat in DLKD priority |
| EXERCISE_BY_ID | Add/remove/modify | New/different exercises available | Add new progression variant |
| BLUEPRINTS | Modify slot_order, families | Different session structure | Add Acc to mandatory families |
| CONDITIONING_PROTOCOLS | Add/retier/modify | Different protocol selected | Promote B-tier protocol to A-tier |
| SUBSTITUTION_MAP | Add cross-family link | Better substitution path | New intent category mapping |
| VALIDATION_RULES | Add/modify check | Different pass/fail outcome | Add neck work check for contact sports |
| EQUIPMENT_MAP | Add/modify equipment list | Better equipment matching | Add "bands" to Basic Gym profile |

## What Cannot Change (Without Architecture Redesign)

| Component | Rationale |
|-----------|-----------|
| 8-step pipeline order | The generation flow is deterministic and validated |
| 15 family codes | Exercise catalog is built around these families |
| 14 blueprint structure | Blueprints are the core abstraction |
| 3 athlete levels | All rulebooks, exercise catalog, and validation built around 3 levels |
| 4 equipment profiles | Equipment matching logic depends on 4 profiles |
| 1-5 difficulty scale | Exercise difficulty anchors depend on this scale |
| 12-point credibility check | Scoring is based on 12 checks |

## Feedback Types

### Type 1: Exercise Substitution Feedback

*"This exercise isn't right for this athlete in this context. I would use X instead."*

**Trigger:** Coach reviews session, swaps an exercise.
**Capture:** Log the (family, original_exercise, substituted_exercise, reason, athlete_profile) tuple.
**Action:**
- If same substitution occurs ≥3 times in similar contexts: adjust selection_priorities for that family
- If substitution is due to equipment: update EQUIPMENT_PROFILE_MAP
- If substitution is due to injury: update injury_substitution table

### Type 2: Structure Ordering Feedback

*"I would put this block earlier/later in the session."*

**Trigger:** Coach reorders blocks.
**Capture:** Log the (blueprint, families, original_order, new_order, reason).
**Action:**
- If consistent reorder discovered: update blueprint.slot_order
- If reorder is specific to a sport: add sport-specific ordering rule to blueprint selection guide

### Type 3: Conditioning Protocol Feedback

*"This conditioning protocol is wrong for this athlete/goal/environment."*

**Trigger:** Coach replaces conditioning protocol.
**Capture:** Log the (athlete profile, original_protocol, new_protocol, reason).
**Action:**
- If same swap happens repeatedly: adjust tier ranking (demote protocol from A to B)
- If protocol is systematically unsuitable for a sport: add sport-specific exclusion rule
- If protocol parameters are consistently adjusted: update level_variants

### Type 4: Missing Content Feedback

*"This session is missing something important."*

**Trigger:** Coach adds an exercise that FORGE did not include.
**Capture:** Log the (blueprint, athlete_profile, added_exercise, family, reason).
**Action:**
- If same exercise is added ≥3 times: promote it in selection_priorities for that family
- If an entire family is added consistently: consider promoting from optional to mandatory in that blueprint

### Type 5: Credibility Failure Feedback

*"This session would not pass my coaching standards."*

**Trigger:** Scorecard total < 6.0.
**Capture:** Full scorecard with per-criterion scores and coach comments.
**Action:**
- Identify which criteria scored lowest
- Determine if it's a data problem (wrong exercise selected), rule problem (missing check), or structural problem (blueprint definition)
- Fix at the appropriate level
